# AWS secret rotation runbook

Operator runbook for rotating Secrets Manager values used by VYU ECS services.

## Scope

This runbook covers rotation for:

| Secret | Path |
| --- | --- |
| Database connection material | `vyu/{env}/database/connection` |
| Provider credentials | `vyu/{env}/providers` |

Use `scripts/configure_secrets.py` for approved `put-secret-value` updates.

## Critical ECS behavior

ECS task definitions inject Secrets Manager values into container environment variables at task launch time.

**Running tasks do not pick up rotated secret values automatically.**

After `put-secret-value`, you must launch new tasks (for example `aws ecs update-service --force-new-deployment`) before assuming the new credential is active in the fleet.

Plan rotation during a maintenance window and verify health after task replacement.

## 1. Prerequisites

- Approved change ticket
- Operator access to Secrets Manager in the target account
- New secret material generated by the provider or database operator
- Overlap window for provider keys when the upstream supports dual-active credentials

Record operator name, UTC start time, secret name, and environment.

## 2. Provider key overlap

For external providers that allow overlapping keys:

1. Issue the new provider key while the old key remains valid.
2. Write the new key to Secrets Manager.
3. Force ECS new deployments so tasks read the new value.
4. Verify application health and provider traffic.
5. Revoke the old provider key only after the overlap window and successful smoke.

For database credentials, coordinate with the DBA for user/password rotation and application reload. Prefer dual-password support only when the database engine and runbook explicitly allow it.

## 3. Update the secret

Never pass secret values on the command line. Read from a local file or stdin.

JSON secret example:

```powershell
@'
{
  "OPENAI_API_KEY": "<new-value>"
}
'@ | uv run python scripts/configure_secrets.py `
  --environment staging `
  --secret-id "vyu/staging/providers" `
  --value-stdin
```

The script verifies the secret `Environment` tag before calling `put-secret-value`.

Capture the returned `version_id` in the rotation record. Do not log secret contents.

## 4. Force ECS new deployment

Rotate tasks for every service that consumes the secret:

```powershell
$env = "staging"
foreach ($service in @("web", "api", "worker")) {
  aws ecs update-service `
    --cluster "vyu-$env" `
    --service "vyu-$env-$service" `
    --force-new-deployment
}

aws ecs wait services-stable --cluster "vyu-$env" --services "vyu-$env-web" "vyu-$env-api" "vyu-$env-worker"
```

If the migration task uses the rotated secret, run a no-op migration verification task only when required by the change plan.

## 5. Health verification

1. Confirm ECS services are stable.
2. Check CloudWatch alarms for auth, provider, and job-failure spikes.
3. Run smoke:

```powershell
$env:VYU_DEPLOY_SMOKE_BEARER_TOKEN = "<short-lived bearer token>"
uv run python scripts/deploy_smoke.py --base-url "https://staging.example.com"
```

4. Perform one authenticated provider-touching action in staging if the rotated secret affects live provider calls.

## 6. Revoke old credentials

After the overlap window and successful verification:

1. Revoke the previous provider key or database password through the upstream control plane.
2. Confirm no tasks still reference the retired material.
3. Mark the Secrets Manager version as deprecated in the change record if your process requires it.

## 7. Audit record

Attach to the change ticket:

```json
{
  "environment": "staging",
  "operator": "<name>",
  "secret_id": "vyu/staging/providers",
  "new_version_id": "<secrets-manager-version-id>",
  "ecs_forced_deployments": [
    "vyu-staging-web",
    "vyu-staging-api",
    "vyu-staging-worker"
  ],
  "overlap_started_at_utc": "<iso8601>",
  "old_credential_revoked_at_utc": "<iso8601>",
  "smoke_status": "pass",
  "notes": "ECS env-injected secrets require new tasks; running tasks retained old value until deployment completed."
}
```

## 8. Abort conditions

Stop rotation and escalate if:

- `configure_secrets.py` rejects the environment tag
- ECS services fail to stabilize after forced deployment
- Smoke or provider verification fails
- Provider overlap window expires before new tasks are healthy

If a bad secret version was written, restore the prior Secrets Manager version through AWS version staging, force new deployments again, and record the incident.

## 9. Staging exercise checklist

Complete once per environment before production rotation:

- [ ] Rotate a non-production test value in `dev`
- [ ] Force ECS redeploy and confirm running task count replaced
- [ ] Capture smoke output and alarm state
- [ ] Revoke the superseded test credential
- [ ] Store evidence with operator name and timestamps

Production rotation additionally requires owner approval and an active rollback contact.
